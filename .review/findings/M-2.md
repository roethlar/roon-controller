# M-2: Play-bar navigation commits history before final browse succeeds

**Severity**: MEDIUM
**Status**: Ready for review (reopen #1 addressed)
**Branch**: `fix/m-2-playbar-history-commit`
**Commit**: see `.review/ready/M-2.json`

## What

`resolveAndNavigate` commits search UI/history before the actual drill-down browse has succeeded. After finding a matching search result, it calls `setSearchLoading`, `resetHistory`, and `pushHistory` at `ui/src/routes/+layout.svelte:319-337`, then performs the browse that populates the pane at `ui/src/routes/+layout.svelte:340-345`. If that second REST call fails, the catch block at `ui/src/routes/+layout.svelte:352-357` only toasts. The store can keep `searchLoading` true, the previous history is gone, and the new history points at an item that never rendered. From non-library routes this also leaves a staged Library history without navigating to `/library`.

## Approach

Same deferred-commit pattern as M-1: move the visible-state mutations (`setSearchLoading` + `resetHistory` + `pushHistory`) to AFTER the drill `apiBrowse` succeeds. The first `apiBrowse` (search) and the second (drill) are now both pure inputs to the decision — neither mutates UI/history state. A failure of either leaves history and search context exactly as they were.

Also captured `onLibrary` at function start (`$page.url.pathname === '/library'`) for symmetry with the M-1 fix and to avoid a stale read if `$page` changes mid-await.

No restore helper needed: with all mutations deferred, the catch block has nothing to undo. Just the toast remains.

The success-path UX is unchanged: setSearchLoading still fires (re-seeds `lastSearchQuery` for Search.svelte's "for X" label and the library page's forward-nav reads), and on `/library` the immediately-following `setBrowseResult` flips `searchLoading` back to false. Off-library, `searchLoading` stays true briefly until `/library` mount's `restoreBrowse` renders the result.

## Known gaps

- The success path leaves `searchLoading=true` momentarily on the off-library route between `setSearchLoading` and `goto('/library')`. This was the pre-existing behavior. The `/library` mount's `restoreBrowse` flow clears it on the next `setBrowseResult`. If `restoreBrowse` were to throw before rendering, `searchLoading` would be stuck — out of scope for M-2 (separate finding worth opening if it's not already covered).
- `lastSearchQuery` is mutated on the success path via `setSearchLoading(opts.input)`. This is preserved from the original. A separate question whether play-bar nav SHOULD seed `lastSearchQuery` (the user didn't search) — but changing that is out of scope here.
- The fix scope is `resolveAndNavigate`. Any other call site that mutates history before a REST drill is its own finding.
