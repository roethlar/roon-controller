# M-4: Queue subscriptions accept unbounded item counts

**Severity**: MEDIUM
**Status**: Ready for review
**Branch**: `fix/m-4-queue-unbounded-counts`
**Commit**: see `.review/ready/M-4.json`

## What

The queue subscription size is validated as positive, but never capped. The REST GET path accepts any positive integer query at `src/server/http/routes/transport.ts:217-220`, the REST POST path forwards any positive `max_item_count` at `src/server/http/routes/transport.ts:248-253`, and the socket path does the same at `src/server/socket/index.ts:297-311`. `TransportService.resolveQueueSubscriptionSize` then returns the max of the request and zone metadata at `src/core/roon/TransportService.ts:705-724` with no upper bound. A client can request a huge queue subscription, forcing large Roon subscriptions, in-memory snapshots, and socket broadcasts.

## Approach

Two-layer defense:

1. **Upstream validators**: REST GET, REST POST, and socket `queue:subscribe` each reject `max_item_count > MAX` with a 400 / error response. The MAX is exposed as a public `static readonly MAX_QUEUE_SUBSCRIPTION_ITEMS = 50_000` on `TransportService` so validators can reference it (and the error message can include the actual limit).

2. **Defense-in-depth clamp** inside `resolveQueueSubscriptionSize`: even if a future internal caller (or a bypassed validator) passes a huge `requestedMaxItemCount` OR `queue_items_remaining` claims a billion items, the value passed to Roon is `Math.min(MAX, Math.max(MIN, requested, basedOnZone))`. The clamp is silent — the upstream validator owns the user-facing error.

Cap chosen at **50,000** (10× the existing 5,000 MIN floor). Real-world queues rarely top a few thousand; 50k headroom covers extreme playlists without exposing the server to a million-item allocation triggered by a malicious or buggy client.

No backward-compatibility concern: previous behavior accepted any positive integer, so existing clients passing reasonable values (< 50k) are unaffected.

## Files changed

- `src/core/roon/TransportService.ts` — added public `MAX_QUEUE_SUBSCRIPTION_ITEMS = 50_000`; wrapped the existing `Math.max(...)` in `Math.min(MAX, ...)` inside `resolveQueueSubscriptionSize`.
- `src/server/http/routes/transport.ts` — added cap check to the GET `/queue/:zoneId` and POST `/queue/subscribe` handlers; returns `{ error: "maxItems must be ≤ 50000" }` (resp. `max_item_count`) with 400.
- `src/server/socket/index.ts` — added cap check to the `queue:subscribe` socket handler; sends the same error message via the existing `queue:error` channel.
- `src/core/roon/__tests__/TransportService.test.ts` — two new tests in the `subscribeQueue` describe: requested cap + zone-derived cap, both verifying the clamp.
- `src/server/http/__tests__/app.test.ts` — two new tests: GET-with-oversized-`maxItems` returns 400, POST-with-oversized-`max_item_count` returns 400.

## Tests added

In `TransportService › subscribeQueue`:

1. **`M-4: caps requested max_item_count at MAX_QUEUE_SUBSCRIPTION_ITEMS`** — calls `subscribeQueue('zone123', 1_000_000_000)`; asserts the call to `subscribe_queue` uses `MAX_QUEUE_SUBSCRIPTION_ITEMS` (50_000), not the billion.

2. **`M-4: caps zone-derived size at MAX even when queue_items_remaining is huge`** — seeds a zone with `queue_items_remaining: 999_999_999`; subscribes without an explicit size; asserts the `subscribe_queue` call uses `MAX_QUEUE_SUBSCRIPTION_ITEMS`.

In `HTTP app routing`:

3. **`M-4: GET /api/transport/queue/:zoneId rejects maxItems > MAX with 400`** — fetches `?maxItems=999999`; asserts 400 and error message references the cap.

4. **`M-4: POST /api/transport/queue/subscribe rejects max_item_count > MAX with 400`** — POSTs `{ max_item_count: 999_999 }`; asserts 400 and error message references the cap.

### Load-bearing verification

Per `.review/README.md`.

- **Service clamp**: temporarily reverted `resolveQueueSubscriptionSize` to `Math.max(MIN, requested, basedOnZone)` (no Math.min wrap). Both new TransportService tests failed (subscribe_queue received 1_000_000_000 / 999_999_999 instead of 50_000). Restored the clamp.
- **REST validator**: temporarily removed the cap check in GET `/queue/:zoneId`. The HTTP GET test failed (got 200 instead of expected 400). Restored.

The socket path's validator load-bearing wasn't separately verified — it's structurally identical to the REST validators (same check, same MAX constant). A socket-level integration test would be the next test to add if the reviewer wants symmetric coverage.

## Known gaps

- Socket validator has no test coverage. Symmetric with REST validators; would be straightforward to add via the existing socket test harness if it exists. Marked as a known gap rather than a blocker.
- The 50,000 cap is heuristic. If real-world deployments observe legitimate queues approaching that size, the cap can be raised without further code changes — it's a single named constant in one place.
- `resolveQueueSubscriptionSize` is now `Math.min(MAX, Math.max(MIN, ...))` which means MIN > MAX would silently invert. Today MIN=5000, MAX=50000 so this isn't a live concern; if MIN grows past MAX a future change can add an invariant check.
- Zone `queue_items_remaining` could legitimately exceed MAX (an actual million-track queue). The cap means we'd subscribe to the first 50k items only. Acceptable tradeoff — UI pagination handles the rest.
