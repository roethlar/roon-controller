# M-3: Recently Played inserts are broadcast before durable persistence

**Severity**: MEDIUM
**Status**: Ready for review (reopen #1 addressed)
**Branch**: `fix/m-3-rp-broadcast-before-persist`
**Commit**: see `.review/ready/M-3.json`

## What

The clear path now waits for durable persistence, but the insert path still accepts and broadcasts entries before the write is known to have committed. `handleNowPlayingImpl` increments the revision, schedules a fire-and-forget persist, then emits `inserted` at `src/core/recently-played/RecentlyPlayedService.ts:453-456`. `schedulePersist` discards the promise at `src/core/recently-played/RecentlyPlayedService.ts:614-616`, and the chained write path intentionally swallows failures for future writes at `src/core/recently-played/RecentlyPlayedService.ts:624-629`. If disk persistence starts failing after startup, clients can apply entries and revisions that are only in memory; after a process restart, the list reverts to the older file while `/health` still reports OK.

## Reopen #1 — addressed

Reviewer (`.review/results/M-3.reopened.md`) caught three issues with the initial fix:

- **P1**: `clear()` was not serialized through `insertChain`. An insert could land between clear's mutation and clear's persist (or vice versa), letting both ops broadcast stamped with the same live `getRevision()` value at emit time.
- **P1**: Insert rollback (on persist failure) could clobber a concurrent clear's revision bump.
- **P2**: GET `/api/recently-played` served the in-memory snapshot that includes mid-flight inserts whose persist hadn't yet committed — clients could observe a revision that later vanished.

**Fix**:

1. **Unified chain**: `clear()` now enqueues `runClear()` onto the SAME `insertChain` as inserts. The two operations FIFO-serialize: any insert that lands before `clear()` runs to completion (including emit/rollback) before `clear`'s `runClear()` starts, and any insert that lands after `clear()` waits for `cleared` to fire before running. Both P1s vanish — they relied on the two ops overlapping.

2. **Coalescing preserved**: `pendingClear` still memoizes the current in-flight clear so concurrent DELETE callers share one chain entry. After completion the field clears itself so the next clear starts a fresh entry.

3. **Removed deferred-buffer machinery**: `clearInFlight` + `pendingDuringClear` + `drainPendingInserts()` were necessary when clear was OFF-chain to prevent insert mutations during clear's await. With everything on `insertChain` they're redundant — the chain itself serializes.

4. **GET awaits flush**: `GET /api/recently-played` calls `await service.flush()` before reading `getEntries()` / `getRevision()`. Every entry in the response is now durable. `degraded` is re-checked after flush in case an in-flight start flipped the flag.

## Approach

Mirror the clear path's durable-or-nothing contract:

1. **Serialize inserts** through a new `insertChain: Promise<void>` so each `handleNowPlayingImpl` run is atomic with respect to the next. Without serialization, the snapshot/mutate/persist/rollback sequence would interleave across awaits — a failed insert's rollback could clobber a later successful insert's mutation.

2. **Convert `handleNowPlayingImpl` to async**: snapshot `entries` + `revision` BEFORE mutating, then `await schedulePersistAsync()`. On success, emit `inserted`. On failure, restore the snapshot and **do not emit** — clients are never told about an entry that won't survive a restart. The log warning carries the entry's title + zone_id (not the full entry) so secrets in a future field don't leak.

3. **Enqueue path** (`handleNowPlaying` + `drainPendingInserts`) pushes onto `insertChain` via a private `enqueueInsert()` helper. Errors from the impl are logged at the chain boundary so a thrown impl doesn't poison the chain for subsequent inserts.

4. **Public `flush()`** method exposed for tests and any future route handler that needs to observe-after-write: `await this.insertChain.catch(() => undefined); await this.writeChain.catch(() => undefined)`.

The clear path is unchanged. Its existing `clearInFlight` + `pendingDuringClear` machinery still buffers inserts during a clear; on drain, those buffered events flow through the new `enqueueInsert` so they too are durable-or-nothing.

## Tests added

In new describe `RecentlyPlayedService › M-3: insert is durable-or-nothing`:

1. **`does NOT emit 'inserted' and rolls back in-memory state when persist fails after startup`** — seeds a committed entry "Committed", then jest-spy-rejects `fs.writeFile` and fires "Lost". Asserts `inserts === ['Committed']` (no `'Lost'` broadcast), `getEntries() === ['Committed']` (no phantom entry), and the on-disk file still reads `['Committed']`. Restores the spy and fires "Recovered"; asserts the chain isn't poisoned (`'Recovered'` commits and broadcasts).

2. **`preserves revision monotonicity across failed inserts`** — commits "A" (revision 1), then a one-shot writeFile rejection during "Failed" → revision rolls back to 1. Subsequent "B" insert bumps to revision 2, not 3 — clients don't see a phantom higher revision.

The rewritten clear test (`drains buffered inserts onto the rolled-back list when persist fails; drained inserts also roll back if their persist fails`) asserts the new combined contract: when a clear fails AND the buffered insert's persist also fails (writeSpy is still rejecting at drain time), neither `cleared` nor `inserted:MidClear` is broadcast, and in-memory state matches the pre-clear list with "Before" only.

## Known gaps

- `flush()` is exposed publicly. Today only tests need it; route handlers don't (the socket-broadcast model is fire-and-forget). If a future REST GET response must reflect the latest insert that just arrived via the listener, callers can `await flush()` first. The JSDoc documents this.
- An insert that arrives DURING a still-failing-disk window will go through the full snapshot/mutate/persist/rollback cycle even though the persist is doomed. That's correct (we don't have a circuit-breaker), but adds one wasted writeFile attempt per insert during an outage. If the disk stays broken for hours, every now-playing event runs this gauntlet. Acceptable — the cost is small, and a circuit breaker would itself be a state-management exercise (when to half-open, etc.). Out of scope for M-3.
- The rollback is in-process only. If the server crashes between `setEntries(next)` and the persist completing, the next start sees the file's last-committed state (correct). No durable WAL needed — the atomic mkdir/writeFile/rename + the in-memory rollback together preserve the invariant "broadcast → durable".
- The eager-persist-at-startup path (degraded mode) is unchanged. M-3 is about post-startup inserts.
