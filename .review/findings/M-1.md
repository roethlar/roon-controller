# M-1: Rail navigation loses current state on REST failure

**Severity**: MEDIUM
**Status**: Ready for review (reopens #1, #2, #3 addressed)
**Branch**: `fix/m-1-rail-rest-failure`
**Commit**: see `.review/ready/M-1.json`

## What

`navigateToRailEntry` mutates the visible Library state before the REST label-walk has succeeded. When already on `/library`, it sets browse loading at `ui/src/routes/+layout.svelte:406-408`, then clears persisted history at `ui/src/routes/+layout.svelte:409`, before the first `apiBrowse` call at `ui/src/routes/+layout.svelte:411-415`. If that call or any later label-walk call fails, the catch block at `ui/src/routes/+layout.svelte:455-460` only emits feedback; it does not restore the old history and does not clear the browse loading flag. A transient REST/Core failure during a rail click can therefore wipe the user's prior Library back stack and leave the pane stuck in loading over the previous content.

## Reopen #1 — addressed

Reviewer caught (`.review/results/M-1.reopened.md`, original) that the initial fix snapshotted only `current` + `hierarchy`. The optimistic `setBrowseLoading('browse')` clears `error`, so a user looking at an error pane (e.g. "Roon Core unreachable") had the error wiped on a failed rail click — falling back to a blank welcome view instead of the prior error pane.

Initial fix added a `BrowseViewSnapshot` covering `current` + `hierarchy` + `loading` + `error`. A new test pinned the contract.

## Reopen #2 — addressed

Reviewer caught (`.review/results/M-1.reopened.md`, second pass) that `setBrowseLoading` also clears `searchLoading` as a side effect. The narrow snapshot covered `error` but not `searchLoading`, so a user with an in-flight search panel could click a rail item, hit a REST failure, and lose the visible "searching" spinner even though rail navigation rolled back.

**Fix**: replaced the narrow `BrowseViewSnapshot` with a full `BrowseStateSnapshot = BrowseState`. `snapshotBrowseState(state)` returns `{ ...state }` (shallow copy of all eight BrowseState fields), and `restoreBrowseState(prev)` does `internalStore.set({ ...prev })`. This is robust against future "set\*Loading"-style helpers that clear additional cross-slice fields — any field clear is undone by restore.

Tradeoff documented in the JSDoc: any unrelated mutation that lands DURING the await window between snapshot and restore will be reverted. Today's call sites gate concurrency via per-flow `*InFlight` flags (e.g. `railNavInFlight`) so this is not a practical issue. If a future call site needs to coexist with concurrent writes, snapshot+restore is the wrong tool.

A new test pins the contract: seed `setSearchLoading('beatles')` (active search panel), trigger a failed rail click, assert `searchLoading === true` and `lastSearchQuery === 'beatles'` after rollback.

## Reopen #3 — addressed

Reviewer (`.review/results/M-1.reopened.md`, third pass) caught that the full-state restore from reopen #2 wipes legitimate concurrent updates. If `setSearchResults(...)` runs via the search socket DURING the rail-navigation REST await (setting `lastSearch=results`, `searchLoading=false`), the subsequent rail-failure rollback writes the stale snapshot back, losing the results and resurrecting the spinner.

**Fix**: replace `restoreBrowseState` with `restoreBrowseStateIfUnchanged(prior, afterMutation)`. Two snapshots are taken: `prior` (before any mutation) and `afterLoadingSnapshot` (immediately after the optimistic `setBrowseLoading`). On rollback, fields are grouped into two slices — **browse** (`current`, `hierarchy`, `loading`, `error`) and **search** (`lastSearch`, `lastSearchQuery`, `searchLoading`, `searchError`) — and each slice is restored only if every field in it still equals `afterMutation`. If ANY field in a slice was independently changed, the whole slice is left alone.

Slice-level granularity is needed because per-field comparison can't disambiguate idempotent writes (e.g. `setBrowseLoading` and `setSearchResults` both write `searchLoading=false`; per-field Object.is would say "no one touched it, restore prior=true"). Treating the slice atomically respects writer intent: `setSearchResults` writes both `lastSearch` AND `searchLoading`, so the `lastSearch` change signals "this whole slice is mine now" and the `searchLoading` restore is correctly skipped.

A new test pins the contract: seed `setSearchLoading('beatles')`, stall the rail `apiBrowse`, fire `setSearchResults(arrivedResults)` mid-await, then reject the rail call. Asserts `lastSearch === arrivedResults` (results survive) and `searchLoading === false` (spinner not resurrected).

## Approach

Two-part fix in `navigateToRailEntry`:

1. **Snapshot prior view before any mutation.** On `/library`, capture `snapshotBrowseView(get(browseStore))` into local `priorSnapshot` at the top of the function. A `restorePriorView()` closure re-applies the snapshot via `restoreBrowseView(priorSnapshot)`.

2. **Defer history + pane mutations until the walk succeeds.** Instead of `resetHistory()` before the first apiBrowse and `pushHistory()` after each drill, collect each successful drill into a local `plan` array. Only after the full label-walk completes do we `resetHistory()` and push all plan entries — atomic commit. A failure mid-walk leaves history untouched.

Three failure paths now call `restorePriorView()`:
- `catch (err)` — any apiBrowse rejection.
- Stale-label early return — when a label is no longer in the parent result, we surface the feedback toast, refresh the rail, and restore.
- Off-library navigation — `priorSnapshot === null`, the closure is a no-op.

`setBrowseLoading('browse')` still fires up-front so the user sees feedback immediately; the restore on failure puts everything back including a prior error banner.

## Known gaps

- Slice granularity means: if one search-pane field is independently updated (e.g. `setSearchResults` writes `lastSearch`), the whole search slice is preserved at its current value. If the user intended to also revert an unrelated search field (e.g. `searchLoading` got optimistically set true elsewhere), that won't roll back. This is correct conservatism: never wipe an independent writer's update, even at the cost of skipping some prior-state restoration.
- The fix is scoped to the rail-click entry point (`navigateToRailEntry`). The play-bar `resolveAndNavigate` flow is M-2's scope and is fixed separately on `fix/m-2-playbar-history-commit` (which uses a different pattern — deferred mutation only, no snapshot — because its drill failure point doesn't pre-mutate state).
- `railNavInFlight` already prevents concurrent rail clicks; the snapshot is therefore single-writer per click.
