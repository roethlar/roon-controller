# M-1: Rail navigation loses current state on REST failure

**Severity**: MEDIUM
**Status**: Ready for review
**Branch**: `fix/m-1-rail-rest-failure`
**Commit**: see `.review/ready/M-1.json`

## What

`navigateToRailEntry` mutates the visible Library state before the REST label-walk has succeeded. When already on `/library`, it sets browse loading at `ui/src/routes/+layout.svelte:406-408`, then clears persisted history at `ui/src/routes/+layout.svelte:409`, before the first `apiBrowse` call at `ui/src/routes/+layout.svelte:411-415`. If that call or any later label-walk call fails, the catch block at `ui/src/routes/+layout.svelte:455-460` only emits feedback; it does not restore the old history and does not clear the browse loading flag. A transient REST/Core failure during a rail click can therefore wipe the user's prior Library back stack and leave the pane stuck in loading over the previous content.

## Approach

Two-part fix in `navigateToRailEntry`:

1. **Snapshot prior view before any mutation.** On `/library`, capture `browseStore.current` and `hierarchy` into local `priorCurrent` / `priorHierarchy` at the top of the function. A `restorePriorView()` closure re-applies the snapshot via `setBrowseResult(priorCurrent, priorHierarchy)` or, if there was no prior result, clears the loading flag with `clearBrowseLoading()`.

2. **Defer history + pane mutations until the walk succeeds.** Instead of `resetHistory()` before the first apiBrowse and `pushHistory()` after each drill, collect each successful drill into a local `plan` array. Only after the full label-walk completes do we `resetHistory()` and push all plan entries — atomic commit. A failure mid-walk leaves history untouched.

Three failure paths now call `restorePriorView()`:
- `catch (err)` — any apiBrowse rejection.
- Stale-label early return — when a label is no longer in the parent result, we surface the feedback toast, refresh the rail, and restore.
- The (rare) case where the snapshot itself was empty falls back to `clearBrowseLoading()`.

`setBrowseLoading('browse')` still fires up-front so the user sees feedback immediately; the restore on failure clears it.

1. **`M-1: REST failure on the first label-walk call restores prior history and pane`** — seeds prior history + `setBrowseResult(priorResult)`, makes the first apiBrowse reject. Asserts history still has `['prior-key']`, `loading === false`, `current === priorResult`. The toast is surfaced.

Asserts history was never partially committed (only `['prior-key']`), and the pane was restored (`loading === false`, `current === priorResult`).

3. **`M-1: stale rail entry (label missing) restores prior pane`** — popAll returns a result without the rail-entry's label. Asserts the stale-label feedback toast fires, history unchanged, pane restored.

## Known gaps

- The snapshot strategy only restores `current` + `hierarchy`. It does not restore `searchLoading`, `searchError`, etc. — those are not touched by the rail-navigation flow, so they remain in whatever state the user last left them.
- The fix is scoped to the rail-click entry point (`navigateToRailEntry`). The play-bar `resolveAndNavigate` flow is finding M-2's scope and is not touched here.
- `railNavInFlight` already prevents concurrent rail clicks; the snapshot is therefore single-writer per click.
