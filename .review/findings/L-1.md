# L-1: Missing Recently Played readiness/health diagnostics

**Severity**: LOW
**Status**: Ready for review
**Branch**: `fix/l-1-rp-health-diagnostics`
**Commit**: see `.review/ready/L-1.json`

## What

The health route reports only a generic OK payload at `src/server/http/routes/health.ts:6-10`, while Recently Played can now enter an explicit degraded mode that makes its routes return 503 at `src/server/http/routes/recently-played.ts:23-28`. There is no operator-facing readiness/diagnostic surface for `ready`, `degraded`, `epoch`, `revision`, or the last persistence error. That makes startup readiness and later persistence degradation visible only through logs or failed RP calls, not through the standard health check.

## Approach

Expand `/api/health` into a real readiness probe with per-subsystem detail:

1. **`createHealthRouter` now accepts the `RecentlyPlayedService`** (optional — preserves the bare-router signature for any future caller that doesn't have one).

2. **Response shape**:
   ```json
   {
     "status": "ok" | "degraded",
     "ready": true | false,
     "timestamp": "...",
     "subsystems": {
       "recently_played": {
         "ready": true | false,
         "degraded": false | true,
         "epoch": <number>,
         "revision": <number>,
         "entry_count": <number>,
         "last_persist_error": { "message": "...", "ts": "..." }   // optional
       }
     }
   }
   ```

3. **HTTP status reflects readiness**: 200 when all subsystems ready, 503 when any subsystem is degraded. Operators can wire load-balancer health checks straight to `/api/health` without parsing the body — though the body now carries the diagnostic detail when they do.

4. **`lastPersistError` tracking** on `RecentlyPlayedService`: each `persist()` failure records `{ message, ts }`; each successful persist clears it. Surfaced via a new `getLastPersistError()` getter. Doesn't gate routes — `isDegraded()` remains the authoritative readiness flag; this is purely operator diagnostic.

The route response is built defensively: if `recentlyPlayedService` is `undefined`, the `subsystems` block omits the `recently_played` key and `ready` stays true. Tests today always wire the service, but a future test harness or alternate entry-point that doesn't have an RP service won't break.

## Files changed

- `src/core/recently-played/RecentlyPlayedService.ts` — added `lastPersistError` field; `persist()` records/clears it; added `getLastPersistError()` getter.
- `src/server/http/routes/health.ts` — `createHealthRouter` now takes optional `RecentlyPlayedService`; response includes `ready`, `subsystems.recently_played`, conditional 503 status.
- `src/server/http/app.ts` — passes `recentlyPlayedService` to `createHealthRouter`.
- `src/server/http/__tests__/app.test.ts` — updated existing `/api/health` test to assert the new shape; added two L-1 tests (503 when degraded; `last_persist_error` surfacing).

## Tests added

In `HTTP app routing`:

1. **Updated `answers /api/health with status ok + per-subsystem diagnostics`** — asserts the new response shape (`ready`, `subsystems.recently_played` with `ready`, `degraded`, `epoch`, `revision`, `entry_count`).

2. **`L-1: /api/health returns 503 + status degraded when recently_played is degraded`** — injects `degraded = true` on the RP service; asserts 503, `status: "degraded"`, `ready: false`, `subsystems.recently_played.degraded === true`.

3. **`L-1: /api/health surfaces last_persist_error when a persist has failed`** — injects a `lastPersistError` directly; asserts the response includes it verbatim. Stays at 200 since `degraded` is false — `last_persist_error` is diagnostic, not gating.

### Load-bearing verification

Per `.review/README.md`. Temporarily reverted the route to always return 200 + `status: "ok"` regardless of subsystem readiness (the old behavior). The L-1 degraded test failed (got 200, expected 503). Restored the readiness-gated response; all 7 HTTP tests pass.

## Known gaps

- `lastPersistError` is only set by the post-startup `persist()` failures (insert path's writes). The startup eager-persist failure that sets `degraded = true` also throws but doesn't currently route through this field — operators can still see degraded state via the `degraded` flag, so this is a small diagnostic gap rather than a blocker. Out of scope for L-1.
- Other subsystems (RoonClient pairing status, TransportService, BrowseService) aren't surfaced in `/api/health`. The finding scope was specifically RP readiness. The new `subsystems` shape is extensible — adding `core_status`, `transport_ready`, etc. is a one-block-per-subsystem extension.
- The `entry_count` field exposes whether the user has data, which is mild info disclosure. The health route isn't authenticated. If operators expose `/api/health` publicly behind a load balancer, the count is visible. Acceptable for the typical home-LAN deployment this app targets; an authenticated operator endpoint could be added later if needed.
- The response timestamp uses `new Date().toISOString()` (wall clock) rather than the service's injectable `now()` clock; not testable for an exact timestamp value. Acceptable for an operator endpoint.
