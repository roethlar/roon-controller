## Findings

### P2 - Full-state rollback can erase search results that arrive during the rail request

`ui/src/routes/+layout.svelte:437` snapshots the entire `browseStore`, `ui/src/routes/+layout.svelte:446` calls `setBrowseLoading('browse')`, and the catch path restores the old full snapshot at `ui/src/routes/+layout.svelte:520-527`. That restores every search field too (`lastSearch`, `lastSearchQuery`, `searchLoading`, `searchError`) via `ui/src/lib/stores/browseStore.ts:154-159`.

The branch comments say this is safe because call sites gate concurrency with `railNavInFlight`, but that flag only gates rail clicks at `ui/src/routes/+layout.svelte:414-415`. Search socket responses are independent: `ui/src/lib/socket/register.ts:126-128` calls `setSearchResults(results)` whenever a `browse:search` result arrives. A reachable interleaving is:

1. User starts a search; `setSearchLoading('beatles')` makes `searchLoading=true`.
2. User clicks a rail item; M-1 snapshots that loading search state and starts the rail REST walk.
3. The search socket result arrives before the rail REST call fails; `setSearchResults(results)` sets `lastSearch=results` and `searchLoading=false`.
4. The rail REST call rejects; `restoreBrowseState(priorSnapshot)` writes the stale snapshot back, losing the results and resurrecting the spinner.

That trades the original "rail failure loses current pane state" bug for an "unrelated search completion is undone by rail failure" bug. The fix needs to avoid restoring fields that changed independently after the snapshot, or use a narrower/conditional rollback. Add a regression test that stalls `apiBrowse`, applies `setSearchResults(...)` while the rail click is in flight, rejects `apiBrowse`, and asserts the newer search results/searchLoading=false survive.
